About This Talk

Exploring conditional statements and their impact on code quality

The Simple if()


if (condition) {
    // do something
}

Common Patterns


// Guard clauses
if (!isValid) {
    return;
}

// Early returns
if (error) {
    throw new Error('Something went wrong');
}

The Problem: Nested Conditions


if (user) {
    if (user.isActive) {
        if (user.hasPermission) {
            // do something
        }
    }
}

Better: Guard Clauses


if (!user) return;
if (!user.isActive) return;
if (!user.hasPermission) return;

// do something

Ternary Operators


// Simple case
const result = condition ? 'yes' : 'no';

// Avoid nesting
const bad = a ? b ? c : d : e;

// Use if/else instead
let good;
if (a) {
    good = b ? c : d;
} else {
    good = e;
}

Switch vs If-Else


switch (type) {
    case 'admin':
        return handleAdmin();
    case 'user':
        return handleUser();
    default:
        return handleGuest();
}

Modern Alternatives


// Object lookup
const handlers = {
    admin: handleAdmin,
    user: handleUser,
    guest: handleGuest
};

const handler = handlers[type] || handlers.guest;
handler();

Optional Chaining


// Before
if (user && user.address && user.address.city) {
    console.log(user.address.city);
}

// After
console.log(user?.address?.city);

Nullish Coalescing


// Before
const value = data !== null && data !== undefined 
    ? data 
    : 'default';

// After
const value = data ?? 'default';

Best Practices

  • Keep conditions simple and readable
  • Use guard clauses for early returns
  • Avoid deep nesting
  • Consider modern alternatives
  • Write self-documenting code
